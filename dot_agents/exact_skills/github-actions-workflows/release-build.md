# Release build + Homebrew publish

Fires on `release: published` (the event emitted by [`release.md`](release.md) after `gh release create`). Builds cross-platform binaries, packages them as tarballs with a checksums file, uploads them to the release, and publishes a Ruby formula to `supermodellabs/homebrew-tap`.

## Tap details

- **Owner**: `supermodellabs`
- **Tap repo**: [`supermodellabs/homebrew-tap`](https://github.com/supermodellabs/homebrew-tap)
- **Formula folder**: `Formula/`
- **Users install via**: `brew install supermodellabs/tap/<formula>`

Formula files are auto-generated by [homebrew-releaser](https://github.com/Justintime50/homebrew-releaser) ã don't hand-edit them in the tap repo, they get overwritten on the next release.

## Trigger

```yaml
on:
  release:
    types: [published]

permissions:
  contents: write
```

Draft releases don't fire `published`, so the build is gated on an intentional release publish.

## Checkout the tag

```yaml
- name: Checkout
  uses: actions/checkout@v6
  with:
    ref: ${{ github.event.release.tag_name }}
```

Always check out the tag, not the default branch. A commit pushed to `main` between tag creation and workflow start would otherwise sneak into the build.

## Cross-compile

```yaml
- name: Setup Deno
  uses: denoland/setup-deno@v2.0.4
  with:
    deno-version: v2.x

- name: Cross-compile all targets
  run: deno task compile:all
```

Whatever toolchain ã Deno, Go (`GOOS`/`GOARCH`), Rust (`cross`), Bun (`--target`) ã produce four binaries named `<binary>-<target>` in a known directory (`bin/` or `dist/`).

## Tarball naming convention

homebrew-releaser expects assets matching this exact pattern (version has **no** leading `v`):

```text
{repo}-{version}-{os}-{arch}.tar.gz
```

Example for `reishi`, tag `v0.4.0`:

```text
reishi-0.4.0-darwin-arm64.tar.gz
reishi-0.4.0-darwin-amd64.tar.gz
reishi-0.4.0-linux-arm64.tar.gz
reishi-0.4.0-linux-amd64.tar.gz
```

The download URL homebrew-releaser constructs:

```text
https://github.com/{owner}/{repo}/releases/download/{tag}/{repo}-{version}-{os}-{arch}.tar.gz
```

## Package step

The tarball must contain the binary at its final installed name (e.g. `rei`), not the per-target name. The cleanest way is to copy each target binary to the canonical name, tar from that name, then remove it:

```yaml
- name: Package tarballs + checksums
  env:
    TAG: ${{ github.event.release.tag_name }}
  run: |
    set -euo pipefail
    for target in darwin-arm64 darwin-amd64 linux-arm64 linux-amd64; do
      cp "bin/<binary>-${target}" bin/<binary>
      tar -czf "<repo>-${TAG}-${target}.tar.gz" -C bin <binary>
      rm bin/<binary>
    done

    sha256sum <repo>-*.tar.gz > "<repo>-${TAG}-checksums.txt"

    echo "### Artifacts" >> "$GITHUB_STEP_SUMMARY"
    ls -la <repo>-*.tar.gz <repo>-*-checksums.txt | tee -a "$GITHUB_STEP_SUMMARY"
```

Note: the reishi workflow uses `${TAG}` directly (which has the `v` prefix) in the tarball name. homebrew-releaser tolerates both with-`v` and without-`v` versions; the docs above describe the canonical no-`v` form. If you hit checksum mismatches, normalize to no-`v`: `VERSION_CLEAN="${TAG#v}"`.

## Upload to release

```yaml
- name: Upload release assets
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    TAG: ${{ github.event.release.tag_name }}
  run: gh release upload "$TAG" <repo>-*.tar.gz "<repo>-${TAG}-checksums.txt"
```

## Publish to Homebrew tap

```yaml
- name: Publish to Homebrew tap
  uses: Justintime50/homebrew-releaser@v3
  with:
    homebrew_owner: supermodellabs
    homebrew_tap: homebrew-tap
    install: 'bin.install "<binary>"'
    target_darwin_amd64: true
    target_darwin_arm64: true
    target_linux_amd64: true
    target_linux_arm64: true
    update_readme_table: true
    github_token: ${{ secrets.HOMEBREW_TAP_GITHUB_TOKEN }}
```

homebrew-releaser clones the tap repo, downloads each platform tarball to compute its sha256, generates a Ruby formula, and pushes it. The formula filename matches the source repo name. `update_readme_table: true` regenerates the project list in the tap README between `<!-- project_table_start -->` / `<!-- project_table_end -->` markers.

### Target platform flags

| Flag | Platform |
| --- | --- |
| `target_darwin_amd64` | macOS Intel |
| `target_darwin_arm64` | macOS Apple Silicon |
| `target_linux_amd64` | Linux x86_64 |
| `target_linux_arm64` | Linux ARM64 |

### Other useful inputs

| Option | Purpose |
| --- | --- |
| `formula_folder` | Defaults to `Formula`. Override if the tap uses a different layout. |
| `depends_on` | Ruby DSL for formula deps, e.g. `"bash" => :build`. |
| `test` | Ruby block for `brew test`, receives `bin`, `testpath`. Example: `assert_match("USAGE", shell_output("#{bin}/<binary> --help"))`. |
| `skip_commit` | `true` for dry-run on first setup ã generates the formula in-job without pushing. |
| `debug` | Verbose logging. |
| `version` | Override auto-detected version (rarely needed). |
| `custom_require` | Add `require_relative` at top of formula (custom download strategies). |
| `commit_owner` / `commit_email` | Override the commit author on the tap. Defaults to a github-actions identity. |

## Required secret

`HOMEBREW_TAP_GITHUB_TOKEN` ã a GitHub PAT with `repo` scope, stored as a repository secret on the **source** repo (not the tap). It needs write access to `supermodellabs/homebrew-tap`.

The default `GITHUB_TOKEN` won't work for cross-repo pushes ã that's why this is a separate PAT.

## First-time setup

1. Confirm `supermodellabs/homebrew-tap` exists with a `Formula/` directory.
2. Create a GitHub PAT with `repo` scope.
3. Add it as `HOMEBREW_TAP_GITHUB_TOKEN` on the source repo.
4. Add the workflow with `skip_commit: true` and `debug: true`.
5. Cut a release; verify the generated formula in the action logs.
6. Flip to `skip_commit: false`; cut another release.
7. Verify locally: `brew tap supermodellabs/tap && brew install <formula>`.

## Troubleshooting

- **Checksum mismatch**: tarball naming doesn't match the expected pattern. Verify `{repo}-{version}-{os}-{arch}.tar.gz` and stay consistent on the `v`-prefix choice.
- **Formula not updating**: `HOMEBREW_TAP_GITHUB_TOKEN` lacks write access to the tap repo, or the secret name is wrong on the source repo.
- **`brew audit` failures on the tap**: ensure semver tags (`v1.2.3`) and valid `install` / `test` blocks.
- **Binary not found after install**: the `install` block references a name that doesn't exist inside the tarball. The `install: 'bin.install "<binary>"'` value must match the file inside the archive exactly (this is why the package step renames per-target binaries to the canonical name before tarring).
- **`update_readme_table` does nothing**: the tap README needs `<!-- project_table_start -->` and `<!-- project_table_end -->` HTML comment markers around the area to regenerate.
